Overview

What Is It?

A competitive real-time multiplayer game built in Unity using the Mirror networking framework. Players move around a shared arena, collect ammo from snow piles, and throw snowballs at each other to reduce their opponent's HP. The project was built for an Online Gaming Technologies module with a focus on demonstrating correct networking architecture.

The game uses a fully server-authoritative model — the server owns all meaningful game state and clients can only request actions, never set state directly. A debug menu accessible with F1 in-game demonstrates each network exploit being rejected in real time.

Architecture

How It Works

CLIENT
  • Reads keyboard input
  • Calls [Command] methods
  • Interpolates remote players
  • Renders local UI
CmdThrow / CmdSendMovement
SyncVar / ClientRpc
SERVER
  • Validates all actions
  • Owns HP, ammo, position
  • Spawns networked objects
  • Applies damage + death

Clients send requests to the server via [Command] methods. The server validates and processes them, then broadcasts results back via [SyncVar] hooks and [ClientRpc] calls. The [Server] attribute on TakeDamage means clients physically cannot call it — Mirror silently drops any attempt.

Features

What Was Built

  • Server-authoritative game loop — HP, ammo, position and death state all owned and validated server-side
  • Threshold-based movement sync that only fires CmdSendMovement when position changes by 0.2+ units, rotation by 5+ degrees, or 100ms has elapsed — eliminating redundant packets
  • Client-side interpolation using exponential decay Lerp/Slerp so remote players move smoothly regardless of frame rate or packet timing
  • Four server-side security patches: teleport rejection with TargetRpc position correction, rapid-fire cooldown, zero-ammo throw prevention, and name-length clamping
  • Full respawn system — server resets state, broadcasts position to all clients, snaps local tracking
  • Snow pile ammo refill via trigger collision, verified server-side with CmdRefillAmmo
  • Death overlay UI driven by RpcOnDeath broadcast, respawn button wired back through CmdRequestRespawn
  • F1 in-game debug menu demonstrating each exploit being rejected with real-time console feedback

Security

Exploit Patches

Four vulnerabilities were identified in the original client-server design and patched server-side:


Attack Vulnerability Patch
Teleport cheat CmdSendMovement accepted any position with no range check Server measures distance from last accepted position. Moves beyond the threshold are rejected and the client is snapped back via TargetRpc
Rapid-fire throw throwCooldown was enforced client-side only — calling CmdThrow directly bypassed it Server maintains _serverNextThrowTime independently. CmdThrow returns early if called before cooldown expires
Zero-ammo throw Server never checked ammo — a client with 0 ammo could still call CmdThrow Server maintains _serverAmmo separately from the client UI counter. CmdThrow is rejected if _serverAmmo is zero
Oversized name CmdSetupPlayer accepted strings of any length, wasting bandwidth and breaking UI Name length is clamped server-side in CmdSetupPlayer — oversized names are rejected with a warning logged to the connection
